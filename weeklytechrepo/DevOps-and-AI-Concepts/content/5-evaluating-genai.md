# Evaluating GenAI Outputs

## Learning Objectives

- Formulate a strict framework for reviewing AI-generated code.
- Analyze AI code for correctness, performance efficiency, and security vulnerabilities.
- Justify the necessity of Human-in-the-Loop (HITL) engineering.

## Why This Matters

When a junior developer writes a pull request (PR), a senior engineer reviews it with deep skepticism, actively searching for logic flaws, unhandled exceptions, or SQL injection vectors before approving it for production.
When an AI generates 400 lines of mathematically complex Python in three seconds, human psychology often defaults to "Automation Bias"—we instinctively biological assume the machine is flawless and blindly accept the code. This is catastrophic. An AI is just another Junior Developer who types incredibly fast but occasionally hallucinates completely fabricated logic. A modern Software Engineer must adopt an ironclad evaluation framework to audit everything the AI produces.

## The Concept

### The Evaluation Framework

You must assume that 100% of the code an LLM generates contains a subtle, critical flaw until proven otherwise.

Every single response generated by a Large Language Model must be subjected to a 4-pillar evaluation:

1. **Correctness (Does it solve the prompt?)**
2. **Efficiency (Does it scale?)**
3. **Security (Does it introduce vulnerabilities?)**
4. **Maintainability (Does it match our corporate standards?)**

#### 1. Correctness (The Hallucination Check)

LLMs are probabilistic. They guess the next most likely token.

- **The Danger:** If you ask an AI to interface with a hyper-niche, highly obscure Python library, it will often confidently fabricate non-existent `.methods()`. If you try to run the code, it immediately crashes with an `AttributeError`.
- **The Evaluation:** You must isolate the AI code and immediately run it against strict Unit Tests. Never deploy AI code without mathematically proving it handles the "Happy Path" and all edge case failures perfectly.

#### 2. Efficiency (O(N) Complexity)

An AI might generate code that perfectly solves the logic test for 10 users.

- **The Danger:** A human developer might notice the AI used a nested `for` loop (O(N^2) complexity) to search a list. If that code is deployed to Amazon and suddenly asked to process 1,000,000 users, the server will melt down and cost the company $50,000 in compute time overnight.
- **The Evaluation:** You must evaluate the AI's Big-O time complexity. If it generates a slow algorithm, you must re-prompt it: *"This nested loop is O(n^2). Refactor this function to use a Hash Map (Dictionary) to achieve O(1) constant time lookups."* The AI will gladly obey and provide the superior architecture.

#### 3. Security (The OWASP Top 10)

LLMs are trained on billions of lines of code written by humans over the last 20 years. Much of that code was terribly insecure.

- **The Danger:** If you ask an AI to write an SQL query to look up a user, it will frequently generate legacy, unparameterized SQL strings (e.g., `SELECT * FROM users WHERE username = '` + input + `'`). If you blindly accept this, you just introduced a fatal SQL Injection vulnerability into your enterprise application.
- **The Evaluation:** You must act strictly as a Security Auditor. Scan the AI's output for standard OWASP vulnerabilities (Hardcoded credentials, missing authentication, insecure direct object references, and injection vectors).

#### 4. Maintainability (PEP 8 / Clean Code)

- **The Danger:** The AI might give you functional code, but it names its variables `x1`, `x2`, and `temp_var`. It might ignore your company's strict formatting standards.
- **The Evaluation:** Run the AI's output through standard linters like `pylint` or `flake8`. Reprompt the AI to enforce naming conventions: *"Rewrite this code using descriptive variable names and ensure strict PEP8 formatting compliance."*

### Human-in-the-Loop (HITL)

Generative AI will not replace Software Engineers. Generative AI will replace Software Engineers who refuse to use Generative AI.
The ultimate safeguard against the probabilistic chaos of LLMs is **Human-in-the-Loop (HITL)** architecture. An AI proposes the code, but an accountable, deeply technical human being must review, formally test, approve, and execute the final deployment. You are the Architect; the AI is merely the infinitely fast typist.

## Summary

- **Automation Bias** is the dangerous psychological tendency for humans to blindly trust machine output, ignoring obvious logic flaws.
- All AI-generated code must face a strict 4-pillar evaluation: **Correctness, Efficiency, Security, and Maintainability**.
- AI frequently introduces critical **Security Vulnerabilities** (like SQL Injection) inherited from bad historical training data.
- **Human-in-the-Loop (HITL)** ensures a biological engineer physically validates all machine output before it reaches a production server.
