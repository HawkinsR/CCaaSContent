# Lab: Agile Sprint Planning

## The Scenario

You are the new Scrum Master for a startup building a "Yelp for Dogs" review app. The CEO has given you a giant, chaotic list of feature requests (the Product Backlog).

Your development team consists of 3 Junior Engineers. You are about to start a 2-week Sprint. Your capacity is **20 Story Points**.

You must take the raw backlog, estimate the effort (Story Points) for each ticket, and decide which tickets make it into the Sprint Backlog.

## The Raw Backlog (`backlog.txt`)

1. Implement User Login via Google OAuth.
2. Fix typo perfectly spelling "pawsome" on the homepage.
3. Allow users to upload 4K video reviews of dog parks.
4. Establish the Postgres Database Schema and ORM connections.
5. Create a CI/CD pipeline in GitHub Actions.
6. Build an AI recommendation engine that suggests parks based on breed.

## Deliverables

1. Navigate to the `starter_code` directory.
2. Create a markdown file named `Sprint_1_Plan.md`.
3. In this file, categorize the tickets you have chosen for this sprint.
4. For each ticket chosen, assign a Story Point value (e.g., 1, 3, 5, 8). Remember, the total cannot exceed 20.
5. Write a brief 1-paragraph justification for *why* you chose those specific tickets and left the others behind. (Hint: Think about dependencies. Can you build an AI engine before you have a database?)

## Definition of Done

- A `Sprint_1_Plan.md` document exists.
- The total story points roughly equal your capacity.
- The justification logically follows Agile prioritization principles (building foundation/infrastructure before shiny UI features).
